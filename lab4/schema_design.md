# Проектирование документной модели MongoDB

## 1. Выбранные коллекции

В системе используются три основные коллекции:

- `users` — пользователи системы;
- `hotels` — отели;
- `bookings` — бронирования.

Такое разбиение выбрано потому, что все три сущности имеют самостоятельный жизненный цикл и активно переиспользуются в разных запросах API.

---

## 2. Коллекция `users`

### Пример документа

```json
{
  "_id": "ObjectId",
  "login": "kirill_sokolov",
  "password_hash": "salt$hash",
  "first_name": "Kirill",
  "last_name": "Sokolov",
  "created_at": "Date"
}
```

### Почему отдельная коллекция

Пользователь существует независимо от отелей и бронирований. Один пользователь может создать много бронирований и, при необходимости, много отелей. Если хранить данные пользователя внутри бронирования, получится лишнее дублирование и станет сложнее искать пользователя по логину.

### Связи

- с `bookings` — через reference `user_id`;
- с `hotels` — через reference `created_by`.

---

## 3. Коллекция `hotels`

### Пример документа

```json
{
  "_id": "ObjectId",
  "name": "Red Square Hotel",
  "city": "Moscow",
  "address": "Tverskaya 12",
  "stars": 4,
  "rooms_total": 28,
  "amenities": ["wifi", "breakfast", "parking"],
  "created_by": "ObjectId",
  "created_at": "Date"
}
```

### Почему отдельная коллекция

Отель — самостоятельная сущность. Его нужно получать списком, искать по городу, изменять независимо от бронирований. Поэтому хранить отели вложенно в документах пользователей или бронирований невыгодно.

### Почему `amenities` — embedded array

Удобства отеля (`amenities`) всегда принадлежат самому отелю и используются только вместе с ним. Для них не нужна отдельная коллекция, поэтому массив строк внутри документа здесь уместнее, чем reference.

---

## 4. Коллекция `bookings`

### Пример документа

```json
{
  "_id": "ObjectId",
  "user_id": "ObjectId",
  "hotel_id": "ObjectId",
  "hotel_snapshot": {
    "name": "Red Square Hotel",
    "city": "Moscow",
    "address": "Tverskaya 12",
    "stars": 4
  },
  "stay": {
    "check_in": "Date",
    "check_out": "Date",
    "guests": 2
  },
  "status": "active",
  "created_at": "Date",
  "cancelled_at": null,
  "status_history": [
    {
      "status": "active",
      "changed_at": "Date",
      "comment": "booking created"
    }
  ]
}
```

---

## 5. Почему здесь и reference, и embedded

### `user_id` и `hotel_id` — references

Для бронирования важно знать, кто его создал и к какому отелю оно относится. Эти сущности живут отдельно, поэтому здесь логично использовать ссылки:

- один пользователь может иметь много бронирований;
- один отель может участвовать во множестве бронирований;
- сами пользователи и отели должны обновляться независимо.

### `hotel_snapshot` — embedded document

В бронирование дополнительно встроен небольшой снимок данных отеля на момент создания брони:

- название;
- город;
- адрес;
- количество звёзд.

Это сделано для сохранения исторической информации. Если позже у отеля изменится название или адрес, старое бронирование всё равно можно показать в том виде, в котором пользователь его оформлял.

### `stay` и `status_history` — embedded documents

Поля `stay` и `status_history` принадлежат только конкретному бронированию и нигде отдельно не используются. Выносить их в отдельные коллекции нет смысла.

---

## 6. Обоснование выбора embedded/references по связям

### Пользователь → Бронирования

Использован **reference** (`user_id`), потому что:

- у пользователя может быть много бронирований;
- бронирования часто читаются отдельно;
- не хочется бесконечно раздувать документ пользователя.

### Отель → Бронирования

Использован **reference** (`hotel_id`), потому что:

- отель связан со многими бронированиями;
- бронирования потенциально быстро растут;
- поиск доступности номера удобнее делать по отдельной коллекции бронирований.

### Отель → Удобства

Использован **embedded array**, потому что:

- удобства маленькие;
- используются вместе с отелем;
- не требуют отдельного жизненного цикла.

### Бронирование → История статусов

Использован **embedded array**, потому что:

- история относится только к одной броне;
- читать её нужно только вместе с документом бронирования.

---

## 7. Индексы

Для ускорения запросов используются следующие индексы:

- `users.login` — уникальный индекс для входа и поиска пользователя;
- `users.first_name + users.last_name` — ускорение поиска по маске имени и фамилии;
- `hotels.city` — ускорение поиска отелей по городу;
- `hotels.created_by` — полезно для служебных запросов;
- `bookings.user_id + created_at` — быстрое получение бронирований пользователя;
- `bookings.hotel_id + status + stay.check_in + stay.check_out` — проверка доступности номеров;
- `bookings.hotel_id + status` — дополнительные выборки по активным бронированиям отеля.
