# Домашнее задание 05. Кеширование и rate limiting на userver

## 1. Контекст

Система бронирования отелей содержит пользователей, отели и бронирования. В этой версии API написан на `userver`, основная база данных — PostgreSQL, а Redis используется для двух задач:

- кеширование часто читаемых ответов;
- хранение счетчиков rate limiting.

Такой вариант ближе к промышленной схеме: PostgreSQL хранит консистентные данные, Redis снимает нагрузку с частых read-запросов и помогает ограничивать слишком частые write-запросы.

---

## 2. Hot paths

Часто выполняемые операции:

| Endpoint | Почему hot path |
|---|---|
| `GET /hotels` | список отелей может открываться на главной странице или в каталоге |
| `GET /hotels/search?city=...` | поиск по городу — основной сценарий сервиса бронирования |
| `GET /users/{user_id}/bookings` | личный кабинет пользователя часто перечитывает список броней |
| `POST /auth/login` | частый endpoint, чувствительный к перебору паролей |
| `POST /bookings` | дорогая операция: проверка доступности и запись в БД |

Потенциально медленные места:

- обращения к PostgreSQL;
- поиск по городу;
- проверка пересечения дат бронирования;
- повторяющиеся запросы списка отелей;
- массовые попытки логина или создания бронирований.

---

## 3. Требования к производительности

Для учебного проекта приняты ориентировочные требования:

| Операция | Целевое время ответа |
|---|---:|
| Ответ из Redis cache | до 50 мс |
| Ответ из PostgreSQL | до 200 мс |
| Создание бронирования | до 300 мс |
| Логин пользователя | до 200 мс |

Основная цель — уменьшить число одинаковых SQL-запросов и защитить систему от частых повторных write-запросов.

---

## 4. Стратегия кеширования

Используется стратегия **Cache-Aside**:

1. Handler сначала проверяет ключ в Redis.
2. Если значение найдено, API сразу возвращает его клиенту.
3. Если значения нет, handler выполняет SQL-запрос к PostgreSQL.
4. Полученный JSON-ответ сохраняется в Redis с TTL.

Причина выбора Cache-Aside: стратегия простая, хорошо подходит для REST API и не требует, чтобы база данных знала о кеше.

---

## 5. Кешируемые endpoints

### `GET /hotels`

- Ключ: `cache:hotels:list:v1`
- TTL: 60 секунд
- Инвалидация: при `POST /hotels`

Этот endpoint читает общий список отелей. Он может часто повторяться и редко требует мгновенно свежих данных.

### `GET /hotels/search?city=...`

- Ключ: `cache:hotels:city:{city}:v1`
- TTL: 120 секунд
- Инвалидация: при создании отеля в соответствующем городе

Поиск по городу — типовой повторяющийся запрос. Например, много пользователей могут искать отели в `Moscow`.

### `GET /users/{user_id}/bookings`

- Ключ: `cache:users:{user_id}:bookings:v1`
- TTL: 30 секунд
- Инвалидация: при создании или отмене бронирования пользователя

TTL меньше, потому что бронирования пользователя меняются чаще, чем общий список отелей.

---

## 6. Инвалидация кеша

Инвалидация сделана в write-endpoints:

| Операция | Какие ключи удаляются |
|---|---|
| `POST /hotels` | `cache:hotels:list:v1`, `cache:hotels:city:{city}:v1` |
| `POST /bookings` | `cache:users:{user_id}:bookings:v1` |
| `PATCH /bookings/{booking_id}/cancel` | `cache:users:{user_id}:bookings:v1` |

В этой работе не используется удаление по маске, чтобы не делать лишний `SCAN` по Redis. Удаляются конкретные известные ключи.

---

## 7. Заголовки кеша

Для кешируемых endpoints добавлены заголовки:

- `X-Cache: HIT` — ответ пришел из Redis;
- `X-Cache: MISS` — был запрос к PostgreSQL, затем результат сохранен в Redis;
- `X-Cache-TTL` — TTL ключа в секундах.

Так можно проверить кеш без профилировщика: два одинаковых `GET /hotels` подряд должны дать сначала `MISS`, затем `HIT`.

---

## 8. Rate limiting

Используется алгоритм **Fixed Window Counter** на Redis:

1. Для endpoint создается ключ вида `rate:{operation}:{subject}`.
2. Каждый запрос делает `INCR`.
3. Первый запрос в окне ставит `EXPIRE` на 60 секунд.
4. Если счетчик больше лимита, API возвращает `429 Too Many Requests`.

Алгоритм выбран потому, что он простой, быстрый и достаточно понятный для учебного проекта.

---

## 9. Лимиты

| Endpoint | Ключ | Лимит |
|---|---|---:|
| `POST /auth/login` | `rate:login:{login}` | 5 запросов в минуту |
| `POST /bookings` | `rate:booking-create:user:{user_id}` | 10 запросов в минуту |
| `POST /hotels` | `rate:hotel-create:user:{user_id}` | 20 запросов в минуту |

---

## 10. Заголовки rate limiting

Для ограничиваемых endpoints возвращаются заголовки:

- `X-RateLimit-Limit`;
- `X-RateLimit-Remaining`;
- `X-RateLimit-Reset`;
- `Retry-After` при превышении лимита.

При превышении лимита возвращается статус:

```http
429 Too Many Requests
```

---

## 11. Как это улучшает производительность

Кеширование уменьшает количество SQL-запросов для повторяющихся чтений. Например, если несколько пользователей открывают список отелей, первый запрос идет в PostgreSQL, а следующие ответы могут быть возвращены из Redis.

Rate limiting не ускоряет одиночный запрос, но повышает устойчивость:

- защищает логин от перебора паролей;
- ограничивает массовое создание отелей и броней;
- снижает вероятность перегрузки PostgreSQL.

---

## 12. Метрики

Для оценки эффективности можно собирать:

- latency p50/p95/p99;
- RPS по endpoint;
- число ответов `2xx`, `4xx`, `5xx`, `429`;
- cache hit rate;
- cache miss rate;
- число SQL-запросов к PostgreSQL;
- количество ключей в Redis;
- средний TTL ключей.

Формула hit rate:

```text
hit_rate = cache_hits / (cache_hits + cache_misses)
```

В учебной версии hit/miss виден по заголовку `X-Cache`.
