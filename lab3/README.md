## Что находится в репозитории

- `schema.sql` — создание таблиц, ограничений и индексов;
- `data.sql` — тестовые данные, не меньше 10 записей в каждой таблице;
- `queries.sql` — основные SQL-запросы для операций из варианта;
- `optimization.md` — объяснение индексов и оптимизации запросов;
- `app/` — API на FastAPI, подключённое к PostgreSQL;
- `Dockerfile` и `docker-compose.yaml` — запуск API и PostgreSQL в контейнерах.

## Предметная область

Система содержит три основные сущности:

### 1. Пользователь (`users`)
Хранит данные для регистрации и поиска пользователей.

Основные поля:
- `id` — первичный ключ;
- `login` — уникальный логин;
- `password_hash` — хэш пароля;
- `first_name`, `last_name` — имя и фамилия;
- `created_at` — дата создания.

### 2. Отель (`hotels`)
Хранит карточку отеля.

Основные поля:
- `id` — первичный ключ;
- `name` — название отеля;
- `city` — город;
- `address` — адрес;
- `stars` — количество звёзд;
- `rooms_total` — число номеров;
- `created_by` — пользователь, который создал запись.

### 3. Бронирование (`bookings`)
Хранит факт бронирования пользователем конкретного отеля.

Основные поля:
- `id` — первичный ключ;
- `user_id` — внешний ключ на пользователя;
- `hotel_id` — внешний ключ на отель;
- `check_in`, `check_out` — даты заезда и выезда;
- `guests` — количество гостей;
- `status` — `active` или `cancelled`;
- `cancelled_at` — дата отмены;
- `stay_period` — вычисляемый диапазон дат для ускорения поиска пересечений.

## Связи между таблицами

- один пользователь может создать много отелей;
- один пользователь может иметь много бронирований;
- один отель может иметь много бронирований.

То есть:
- `users 1:M hotels`
- `users 1:M bookings`
- `hotels 1:M bookings`

## Ограничения и проверки

В схеме использованы следующие ограничения:
- `PRIMARY KEY` для каждой таблицы;
- `FOREIGN KEY` для связей между сущностями;
- `UNIQUE` для логина пользователя;
- `NOT NULL` для обязательных полей;
- `CHECK` для:
  - длины текстовых полей;
  - диапазона количества звёзд от 1 до 5;
  - положительного числа комнат;
  - корректного числа гостей;
  - корректного порядка дат `check_out > check_in`;
  - согласованности полей `status` и `cancelled_at`.

## Индексы

В работе добавлены индексы под реальные операции:

- уникальный индекс по `users.login` создаётся автоматически ограничением `UNIQUE`;
- индекс по `hotels.created_by`;
- индекс по `bookings.hotel_id`;
- составной индекс `bookings(user_id, created_at DESC)`;
- trigram-индекс по полному имени пользователя;
- trigram-индекс по городу отеля;
- частичный GiST-индекс по активным бронированиям и диапазону дат.

Подробное объяснение есть в `optimization.md`.

## Какие API операции поддерживаются

- создание нового пользователя;
- поиск пользователя по логину;
- поиск пользователя по маске имени и фамилии;
- создание отеля;
- получение списка отелей;
- поиск отелей по городу;
- создание бронирования;
- получение бронирований пользователя;
- отмена бронирования.

## Подключение API к PostgreSQL

В прошлой домашней работе использовался SQLite и запросы с плейсхолдерами `?`.
В этой работе API было адаптировано под PostgreSQL:
- вместо SQLite используется `psycopg`;
- запросы переписаны на PostgreSQL-плейсхолдеры `%s`;
- для вставки новых строк используется `RETURNING`;
- проверка пересечения бронирований переписана через `daterange`.

## Пример проверки запросов вручную

```sql
SELECT * FROM users;
SELECT * FROM hotels;
SELECT * FROM bookings;
```

Проверка поиска по городу:

```sql
SELECT id, name, city
FROM hotels
WHERE lower(city) LIKE '%' || lower('moscow') || '%';
```

Проверка бронирований пользователя:

```sql
SELECT *
FROM bookings
WHERE user_id = 3
ORDER BY created_at DESC;
```
